### Added

- **Security envelope around the heartbeat-PR-agent flow** (2026-04-22). Four defence-in-depth controls landed the same day as the heartbeat cutover:
  - **New PreToolUse Bash hook** at `samples/scripts/security/check_bash_command.py`. Closes the Bash-gap in the Edit/Write protection hook — shell-level filesystem operations (`mv`, `cp <dest>`, `rm`, `>` / `>>`, `sed -i`, `tee`, `touch`, `chmod`, `chown`, `truncate`) no longer bypass the protected-path list. Also bans dangerous git ops: pushes to `main` / `master`, any force-push, `git reset --hard origin/main`. Allowlists paths under `samples/` so mirror operations on sample copies of protected files proceed. Fails open on script errors — a bug here never bricks Bash. Python/Node processes that open files internally remain out of scope (hook inspects shell command text only). Hook config added to `samples/.claude/settings.example.json`.
  - **HEARTBEAT.md gained rules 14–16** and a new `## Security envelope` section naming the four controls as load-bearing. Rule 14: never `git push origin main/master`, no force-push. Rule 15: cap the review queue at 10 items — if capacity reached, pause `has-default` builds and classify everything as `needs-intent` until the user drains the queue. Rule 16: honour the `tasks/.heartbeat-dry-run` marker — while present, `has-default` classifications log decisions to `tasks/heartbeat-dry-run.log` without actually building. `needs-intent` and `out-of-scope` classifications continue posting questions as normal during dry-run.
  - **New helper `samples/scripts/heartbeat/review_queue.py`** — counts `pending` + `reminded` entries under `## Active reviews` in `HEARTBEAT_REVIEWS.md`. Exits 0 if under cap (10), 1 if at/over cap. Used by the heartbeat at cycle start to enforce Rule 15.
  - **`create_staging.py` slug validation** — refuses forbidden slugs (`main`, `master`, `develop`, `head`, `origin`, empty), remote-prefixed slugs (`origin/*`, `upstream/*`), path traversal (`../`, `./`), and anything containing `/`. Defence-in-depth against a misfiring classifier producing a worktree branch that collides with the default branch.
  - Rationale: the build-first flow increases the heartbeat's blast radius (it's now producing artifacts rather than asking questions), so the envelope of enforceable controls needs to grow in step. The Bash hook specifically closes a gap discovered during the same-day cutover when the heartbeat.md rewrite completed via `bash mv` against a file hook-protected only against `Edit`/`Write`.

- **Heartbeat-as-PR-agent architecture cutover** (2026-04-22). The heartbeat's workflow shifted from *question-then-action* to *classify-then-act*. Every new task is now classified into `has-default` / `needs-intent` / `out-of-scope` via a deterministic keyword classifier that first greps a durable rejection log to avoid repeating rejected approaches. `has-default` tasks are built speculatively in a sandbox — git worktree for git-scope work, staging folder otherwise — and parked in a review queue (`tasks/HEARTBEAT_REVIEWS.md`) for the user to integrate or reject. Rejections append an ADR-style block to `tasks/HEARTBEAT_REJECTIONS.md` (Attempted / Rejected because / Lesson for future attempts); three matches on a task force `needs-intent` classification (circuit breaker). Four new primitives under `samples/scripts/heartbeat/`: `classify_task.py` (stdlib-only regex heuristics), `check_rejections.py` (rejection-log grep + circuit-breaker status), `create_staging.py` (worktree-or-folder auto-selection based on `git rev-parse`), `__init__.py`. Two new scaffolded task files (`samples/tasks/HEARTBEAT_REVIEWS.md` + `HEARTBEAT_REJECTIONS.md`). `samples/tasks/HEARTBEAT.md` replaced with the full v2 — classify-then-act procedure, per-task-type staging recipe table, review surface write-on-completion + rejection-logging procedure, rejection-history reference, extended sandboxed-execution section with mtime-check for concurrent-edit protection, rules 1–13 including the build-first bias + circuit-breaker enforcement. Rationale: the prior question-first model bottlenecked on user input, producing fatigue (21 open questions accumulated with several 5–7 days old). Build-first uses concrete artifacts as the decision surface — reviewing a diff is faster than answering five open-ended questions. Pattern reference: Architecture Decision Records (ADR — Michael Nygard) for the rejection log; analogous to Dependabot's ignored-versions list.

### Changed
- **Morning-brief scheduled-task root cause fixed (`LogonType: Interactive only` → `Interactive/Background`).** A multi-day mystery — the daily brief appeared "delivered" but `schtasks /Query` showed every firing had actually crashed with `Last Result: 0xC000013A` (`STATUS_CONTROL_C_EXIT`) before producing any output; `scheduled-logs/` was empty for every run. Root cause: the `claude --print` CLI (Node.js) needs a real console handle to manage stdio. When a Task Scheduler task fires with `LogonType: Interactive only` (the default when a task is created without selecting "Run whether user is logged on or not") the PowerShell process launches in a detached/hidden session with no attached console; Node's stdio handshake fails and the process dies on a broken-pipe console signal before writing a byte — the wrapper's `Tee-Object` never gets any data to write. Fix: each Claude task's principal flipped via Task Scheduler → Properties → General → "Run whether user is logged on or not" + Windows password, which changes `LogonType` to `Password` (shown as `Interactive/Background` in `schtasks /Query`) and gives the task a proper batch-logon session with a valid console. Immediate effect — the first post-fix firing produced `Last Result: 0` and the expected log + brief file. Applied to all three live Claude tasks (`Morning Brief`, `Consolidate Memory`, `Heartbeat Monitor`). Documented in `META_ARCHITECTURE.md` §3 Automated infrastructure as a critical-setting note. (2026-04-21)

### Changed
- **`link-check` workflow split into two modes** — on pull requests, lychee now checks only the `.md` files actually changed in the diff (via `git diff --name-only --diff-filter=ACMR`); on the weekly cron and manual dispatch it still runs against the full repo and auto-opens an Issue on failure. Rationale: whole-repo checks on every PR made contributors responsible for link rot they hadn't introduced, and flooded the maintainer's inbox with failure emails on unrelated PRs. Diff-only scopes PR failure to the author's actual change; the weekly cron catches background rot (stale vendor URLs, dangling internal links after deletions) on a reviewable rhythm. Pattern is standard in larger OSS repos (kubernetes, rust, etc.). No third-party action added for the diff — plain `git diff` in a bash step. (2026-04-21)
